# Поиск статей справки по пользовательскому вопросу

## Контекст

Пользователи Авито часто обращаются в поддержку с короткими вопросами: как
отправить товар, почему объявление всё ещё на проверке, где посмотреть статус
заказа, как вернуть деньги и так далее.

В продуктовой системе статьи справки не используются как готовый ответ
пользователю. Они служат документами для RAG-пайплайна: сначала нужно найти
подходящие материалы, а уже потом на их основе формируется ответ.

В этой задаче нужно решить этап поиска: по тексту вопроса пользователя найти
статьи, которые стоит передать дальше в пайплайн.

## Задача

Для каждого запроса из `test.f` верните ранжированный список `article_id`:
сначала самые подходящие документы, затем менее подходящие.

В одном запросе может быть несколько правильных статей. Например, пользователю
может понадобиться статья с инструкцией и отдельная статья с правилами или
условиями сервиса. Поэтому важно не только найти одну наиболее похожую статью,
но и расположить все релевантные документы как можно выше в топ-10.

## Данные

Вам доступны три файла в формате Feather:

- `articles.f` — база статей справки.
- `calibration.f` — размеченные запросы для локальной проверки подхода.
- `test.f` — запросы, для которых нужно подготовить ответ.

### `articles.f`

Колонки:

- `article_id` — целочисленный идентификатор статьи.
- `title` — заголовок статьи.
- `body` — текст статьи в HTML.

Текст статей дан в том виде, в котором он хранится в базе: с HTML-разметкой,
таблицами, вкладками, спойлерами и другими служебными элементами. Выберите
подходящий способ обработки самостоятельно.

### `calibration.f`

Колонки:

- `query_id` — идентификатор запроса.
- `query_text` — текст вопроса пользователя.
- `ground_truth` — правильные `article_id`, разделённые пробелами.

Пример:

```text
query_text: Здравствуйте,подскажите как мне отправить кроссовки покупателю через пункт «Авито»?
ground_truth: 1909 4396
```

Это значит, что для запроса в RAG-пайплайн нужно передать статьи `1909` и
`4396`.

### `test.f`

Колонки:

- `query_id` — идентификатор запроса.
- `query_text` — текст вопроса пользователя.

Для этих запросов нужно сформировать файл с ответами.

## Формат решения

Нужно отправить файл `answer.csv` в формате CSV.

В файле должны быть колонки:

- `query_id` — тот же идентификатор, что в `test.f`.
- `answer` — строка с ранжированным списком `article_id`, разделённых пробелами.

В `answer.csv` должны быть ответы для всех `query_id` из `test.f`: без
пропусков, лишних строк и повторов. Для каждого запроса можно вернуть не больше
10 статей. Повторять один и тот же `article_id` внутри одного ответа нельзя.
Все `article_id` должны существовать в `articles.f`. Нужно возвращать именно
идентификаторы статей, а не ссылки, названия статей или идентификаторы разделов.
Если у решения есть ранжированный список кандидатов, обычно имеет смысл
возвращать до 10 статей: дополнительные документы в конце списка не ухудшают
метрику, если порядок уже выбранных документов не меняется.

Пример:

```text
query_id answer
1        1909 4396 4403 2693 4433 2408 4219 4218 4217 4216
2        1730 1746 1747 1774 1775 1776 1784 1809 1824 1825
3        1827 1828 1829 1830 1831 1833 1834 1835 1836 1837
```

Порядок важен: документ на первой позиции считается более релевантным, чем
документ на второй.

Минимальный пример записи файла:

```python
import pandas as pd

test = pd.read_feather("test.f")

# Здесь должен быть ваш код ранжирования.
# В примере для всех запросов возвращается один и тот же набор статей.
answer = test[["query_id"]].copy()
answer["answer"] = "1909 4396 4403 2693 4433 2408 4219 4218 4217 4216"

answer.to_csv("answer.csv", index=False)
```

## Метрика

Решения оцениваются по `MAP@10`.

Для одного запроса считается `AP@10`: чем выше в списке стоят релевантные
документы, тем больше их вклад. Затем значение усредняется по всем запросам.

Если для запроса есть несколько правильных статей, в оценке учитываются все
найденные документы, а не только первый.

## Ограничения

- Нельзя использовать внешние LLM/API и любые платные или удалённые сервисы для
  получения ответов на тестовые запросы.
- Можно использовать локальные библиотеки и локальные open-source модели.
- Нельзя размечать тестовые запросы вручную или подбирать ответы под конкретные
  `query_id`.
- Нельзя использовать данные, которых нет в выданных файлах, если они раскрывают
  ответы на тестовую часть.
- Решение должно быть воспроизводимым: приложите код, который при повторном
  запуске получает ваш `answer.csv`.

## Что сдавать

Нужно отправить:

- `answer.csv` — файл с ответами для `test.f`.
- Ссылку на код решения или архив с кодом.
- Краткое описание подхода: как вы обрабатывали статьи и запросы, какие модели
  или алгоритмы использовали, как проверяли качество на `calibration.f`.

Лучшие решения будут дополнительно проверяться вручную. При проверке будет
важно, что решение обрабатывает задачу в общем виде, а не задаёт ответы для
отдельных запросов.

## Подсказки

Эта задача про реальные тексты поддержки, поэтому в данных есть шум:
пользовательские формулировки отличаются от языка статей, в запросах бывают
опечатки и разговорные выражения, а HTML статей содержит служебную разметку.

Рекомендуем начать с базового решения и анализа ошибок на `calibration.f`: для
каких запросов статьи не находятся, какие статьи путаются между собой, как
влияет очистка HTML и как выбирать несколько статей для одного запроса.
